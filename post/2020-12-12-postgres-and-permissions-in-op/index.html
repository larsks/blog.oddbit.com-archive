Postgres and permissions in OpenShift

Published Sat, Dec 12, 2020 by Lars Kellogg-Stedman

Folks running the official postgres image in OpenShift will often encounter a problem when first trying to boot a Postgres container in OpenShift. Given a pod description something like this:

apiVersion: v1
kind: Pod
metadata:
  name: postgres
spec:
  containers:
    - name: postgres
      image: postgres:13
      ports:
        - containerPort: 5432
      volumeMounts:
        - mountPath: /var/lib/postgresql/data
          name: postgres-data
      envFrom:
        - secretRef:
            name: postgres-secret
  volumes:
    - name: postgres-data
      persistentVolumeClaim:
        claimName: postgres-data-pvc

The container will fail to start and the logs will show the following error:

chmod: changing permissions of '/var/lib/pgsql/data': Operation not permitted

You can resolve this error by modifying the mountPath to mount the volume on /var/lib/postgresql instead. What’s going on here?

The first thing we need to understand is that containers run with different permissions in an OpenShift environment than they do in Docker. In Docker, a container will by default run as root, which means container startup scripts can make arbitrary changes to the container filesystem. This isn’t true for a default OpenShift environment; in this case, a container runs under a random unprivileged user id (similar to what you would get if you were to run docker run -u $RANDOM ...). That means container scripts and applications can only write to directories for which they have explicit write access. The consequence is that, in general, they won’t be able to write anywhere in a container filesystem…unless you’ve mounted a volume.

When OpenShift mounts a volume into a container, the mountpoint is owned by the root user, but the group is set to the GID corresponding to the random user id under which the container is running. That is, if I mount a volume onto /data, then the permissions on /data will look something like this:

drwxrwsrwx. 3 root 1000690000 1 Dec 12 15:21 /data

So your container will be able to create files and directories under /data, but nowhere else.

Next, we need to look at the startup scripts in the postgres image. We can find these in the postgres image github repository. In particular, we’re interested in the ENTRYPOINT script, which for the postgres:13 image is docker-entrypoint.sh. We’re interested in lines 39 and 40 (note that $PGDATA defaults to /var/lib/postgresql/data):

mkdir -p "$PGDATA"
chmod 700 "$PGDATA"

This is both the source of the error message and the reason why changing mountPath resolves the problem.

First, consider the behavior when we are using mountPath: /var/lib/postgresql/data:

  1. The mkdir -p command does nothing and succeeds, because /var/lib/postgresql/data already exists.

  2. The chmod 700 command fails, because while are current user id has write access to the target directory, it does not own the target directory. Under Linux, you must own a file or directory in order to manipulate its permissions.

Now, consider the behavior when we modify the pod description to use mountPath: /var/lib/postgresql instead:

  1. The mkdir -p command creates /var/lib/postgresql/data, because (a) it doesn’t exist and (b) we have write access to /var/lib/postgresql.

  2. The chmod 700 command succeeds because, since we created /var/lib/postgresql/data, we own it.

And now you know.
